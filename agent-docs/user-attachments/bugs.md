## BUG-001: Backend accepts `http://` links — spec requires `https://` only

- **Severity**: Medium
- **Type**: Security / Functional
- **Component**: `src/handlers/tasks.js` — `validateAttachments()`, constant `LINK_SCHEME_ALLOWLIST`
- **OWASP Reference**: A05:2021 Security Misconfiguration

### Reproduction Steps

1. Bypass the frontend (e.g., curl).
2. `PATCH /api/v1/spaces/:spaceId/tasks/:taskId/attachments` with:
   ```json
   {
     "attachments": [
       { "name": "Insecure Link", "type": "link", "content": "http://insecure.example.com", "author": "user" }
     ]
   }
   ```
3. Observe: `200 OK` — attachment stored and returned.
4. Open the TaskDetailPanel — the http:// link renders as a clickable `<a>` element.

### Expected Behavior

HTTP (non-TLS) links should be rejected by the backend with `400 VALIDATION_ERROR` matching the spec error message: `"A link attachment must have a valid https:// URL as its content."`

### Actual Behavior

The backend accepts `http://` links. `LINK_SCHEME_ALLOWLIST = ['http:', 'https:']` allows both schemes. The frontend correctly enforces `https://` only via `validateContent()`, but API bypass circumvents this guard.

### Root Cause Analysis

`LINK_SCHEME_ALLOWLIST` in `src/handlers/tasks.js` (line 36) was set to `['http:', 'https:']` instead of `['https:']`. The error message emitted by the backend (`"must be a valid http(s) URL"`) also diverges from the spec (`"A link attachment must have a valid https:// URL as its content."`).

### Proposed Fix

1. Change `LINK_SCHEME_ALLOWLIST` from `['http:', 'https:']` to `['https:']`.
2. Update the error message to match the spec: `"A link attachment must have a valid https:// URL as its content."`.
3. No frontend changes needed (already correctly enforces https://).

## BUG-002: PATCH merge does not protect agent attachments from API-level overwrites

- **Severity**: Medium
- **Type**: Functional / Security
- **Component**: `src/handlers/tasks.js` — `handlePatchAttachments()`

### Reproduction Steps

1. Create a task with an agent attachment (e.g., `name: "ADR-1.md"`, `type: "file"`, no `author` field).
2. Call PATCH directly (bypass frontend):
   ```json
   {
     "attachments": [
       { "name": "ADR-1.md", "type": "link", "content": "https://evil.com", "author": "user" }
     ]
   }
   ```
3. Observe: `200 OK`. The agent attachment `ADR-1.md` is now replaced with `type: link, content: https://evil.com, author: user`.
4. Open the TaskDetailPanel — the agent artifact is gone, replaced by the user's link.

### Expected Behavior

Per acceptance criterion: "el merge por nombre no pisa los artefactos que escribe el pipeline." Agent-authored attachments should not be replaceable via the PATCH endpoint, regardless of whether the call comes from the UI or a direct API client.

### Actual Behavior

`handlePatchAttachments` uses a blind `Map`-based upsert that replaces any existing attachment with the same name, regardless of the existing attachment's `author` field. There is no ownership check in the merge logic.

**Note:** The `api-spec.json` explicitly documents this gap: _"The `author` field is stored and returned but not validated for ownership at this layer (backend relies on the DELETE guard for security)."_ This was a design decision by the UX/API stage, but it conflicts with the stated acceptance criterion.

### Root Cause Analysis

The merge loop in `handlePatchAttachments`:

```js
for (const a of existingAttachments) {
  merged.push(incomingByName.has(a.name) ? incomingByName.get(a.name) : a);
  seen.add(a.name);
}
```

does not check `a.author` before replacing. If `incomingByName` has an entry with the same name as an agent attachment, the agent attachment is silently overwritten.

### Proposed Fix

Add an ownership check in the merge loop: if the existing attachment has `author !== 'user'` (i.e., agent-owned or unknown provenance), the incoming payload should NOT replace it; instead, return a `409 Conflict` with a message like `"Attachment 'ADR-1.md' is a pipeline artefact and cannot be overwritten."` This would make the PATCH endpoint consistent with the DELETE endpoint's protection semantics.

Alternatively (lighter fix): filter incoming attachments to exclude names that already exist with `author !== 'user'`, preserving the upsert semantics for user-owned attachments only.

### KNOWN-001: `terminal.test.js` and `pipeline-templates.test.js` hang on exit (exit code 144 — SIGKILL)

- **Severity**: Medium (DevEx / CI)
- **Type**: Test Infrastructure
- **Component**: `tests/terminal.test.js`, `tests/pipeline-templates.test.js`
- **Status**: Pre-existing — unrelated to QOL-7. Excluded from QA test run.
- **Root Cause**: PTY processes and test servers started within these files are never closed, preventing the Node.js process from exiting cleanly. The runner kills them via SIGKILL after timeout.
- **Proposed Fix**: Add `after(() => server.close())` / PTY cleanup in `afterEach`/`after` hooks.
